# Monitoring API

Date: 2026-04-10

## API layers

Нужны два уровня API.

### 1. Agent API

Это `runtime-helper` и его расширения.

### 2. Central Monitoring API

Это новый API для UI, automation и operator tooling.

Ни UI, ни automation не должны ходить по fleet напрямую в helpers как baseline.

## Agent API requirements

### Existing endpoints to reuse

- `GET /v1/helper/status`
- `GET /v1/helper/health`
- `GET /v1/helper/stats.read`
- `GET /v1/helper/events`
- `POST /v1/helper/diagnostics.export`

### Required additions

- `GET /v1/helper/links`
- `GET /v1/helper/links/<linkID>`
- `GET /v1/helper/links/health.stream`
- `POST /v1/helper/links/<linkID>/reconnect`
- `POST /v1/helper/links/<linkID>/drain`
- `POST /v1/helper/links/<linkID>/resume`
- `POST /v1/helper/links/<linkID>/gateway.select`

## Central API design

Base path:

- `/v1/monitor`

## Read APIs

### `GET /v1/monitor/summary`

Returns fleet summary:

- links total
- healthy/degraded/failed/stale counts
- incidents open
- nodes stale
- top failing gateways

### `GET /v1/monitor/links`

Main fleet table endpoint.

Query params:

- `health`
- `state`
- `node_id`
- `gateway_id`
- `role`
- `stale`
- `flapping`
- `q`
- `page`
- `per_page`

### `GET /v1/monitor/links/<linkID>`

Returns current link detail.

### `GET /v1/monitor/links/<linkID>/events`

Returns paginated timeline.

### `GET /v1/monitor/links/<linkID>/probes`

Returns recent probe history.

### `GET /v1/monitor/nodes`

Returns node inventory and health.

### `GET /v1/monitor/nodes/<nodeID>`

Returns node detail:

- links on node
- helper freshness
- current incidents

### `GET /v1/monitor/gateways`

Returns gateway-level rollup.

### `GET /v1/monitor/gateways/<gatewayID>`

Returns gateway detail:

- bound links
- health distribution
- current routing pressure

### `GET /v1/monitor/incidents`

Query params:

- `status`
- `severity`
- `kind`
- `owner`
- `page`
- `per_page`

### `GET /v1/monitor/incidents/<incidentID>`

Returns incident detail with affected links and timeline.

## Live APIs

### `GET /v1/monitor/stream/summary`

SSE for top dashboard counters.

### `GET /v1/monitor/stream/links`

SSE for link changes matching current filters.

### `GET /v1/monitor/stream/incidents`

SSE for incident opens/updates/resolutions.

## Command APIs

### `POST /v1/monitor/commands`

Request body:

- `target_type`
- `target_id`
- `command_type`
- `args`
- `reason`
- `requested_by`
- `idempotency_key`

Response:

- `command_id`
- `status`

### `GET /v1/monitor/commands/<commandID>`

Returns command state and execution trail.

## Command routing rules

### Link-scoped commands

- target helper by `node_id`
- validate lease policy
- create command row first
- then dispatch

### Bulk commands

Bulk should be separate:

- `POST /v1/monitor/commands/bulk`

Because result shape and rollback semantics differ.

## Response conventions

### Current-state endpoints

Should return normalized operator-friendly fields, not raw helper payloads only.

### Detail endpoints

Should include both:

- normalized fields
- raw payload fragments where useful

### Errors

Need explicit error classes:

- `not_found`
- `invalid_filter`
- `stale_target`
- `dispatch_failed`
- `lease_conflict`
- `command_timeout`

## Integration with current control-plane API

### `control-api`

Best path for v1:

- add monitoring API as sibling service or module
- keep peer/uplink routing APIs separate from monitoring APIs

### `portal-http`

Should consume central monitor API, not helper API directly.

Exception:

- temporary canary/debug pages if needed
