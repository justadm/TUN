# Monitoring Backend

Date: 2026-04-10

## Backend goal

Сделать monitoring backend, который:

- принимает telemetry от helpers
- хранит current state и history
- вычисляет health and incidents
- исполняет operator commands
- не ломает существующий `control-plane`

## Recommended code layout

### New directories

- `control-plane/monitoring-ingestor`
- `control-plane/monitoring-api`

Если нужен общий код:

- `control-plane/monitoring-common`

## Service 1: monitoring-ingestor

### Responsibility

- discovery of helper endpoints
- SSE subscription
- polling fallback
- payload normalization
- write to monitoring tables
- freshness tracking

### Internal modules

#### `sources.py`

Knows how to load the set of monitored nodes and helper endpoints.

Sources may be:

- static env/config
- DB inventory
- generated service registry file

#### `helper_client.py`

Implements:

- helper auth
- `GET /links`
- `GET /status`
- `GET /events`
- timeout and retry policy

#### `normalizer.py`

Transforms helper payloads into canonical records:

- `link`
- `snapshot`
- `event`

#### `snapshot_writer.py`

Responsible for:

- upsert current snapshot
- append event rows
- mark stale/seen timestamps

#### `freshness.py`

Determines:

- helper stale
- node unreachable
- delayed ingestion

### Runtime model

The ingestor should run one worker per node with bounded concurrency.

It should prefer SSE, but degrade to periodic polling if:

- SSE unavailable
- connection unstable
- auth rejected transiently

## Service 2: monitoring-api

### Responsibility

- serve read APIs
- serve SSE for UI
- accept commands
- expose incidents and summaries

### Internal modules

#### `queries.py`

Optimized SQL for:

- summary
- links
- nodes
- gateways
- incidents

#### `commands.py`

Validates:

- target existence
- command-type compatibility
- idempotency key
- operator permissions

Writes into `monitor_commands`.

#### `dispatcher.py`

Picks pending commands and calls helper APIs.

#### `incidents.py`

Correlates alerts into incidents.

#### `streams.py`

Publishes SSE based on DB changes or periodic refresh.

## Dispatcher design

### Why separate dispatcher logic

Command execution is different from HTTP request handling:

- remote network calls
- retries
- timeouts
- audit
- asynchronous completion

### Dispatcher loop

1. read pending command
2. lock command row
3. resolve node/helper target
4. call helper endpoint
5. persist result
6. emit command event

### Retry policy

Allowed retries only for:

- transient transport failure
- helper timeout
- temporary 5xx

No retry for:

- validation error
- unknown link
- explicit lease conflict

## Alert evaluator

### Placement

Can live inside `monitoring-api` initially.

### Inputs

- current snapshots
- recent events
- freshness records
- recent probes

### Outputs

- incident open
- incident update
- incident resolve

## Suggested v1 execution model

### Ingest cadence

- SSE primary
- snapshot poll every 15-30s fallback

### Health recomputation cadence

- every 10s for changed links
- every 60s global sweep

### Incident sweep cadence

- every 30s

### Command dispatcher cadence

- 1-2s polling loop

## Auth and permissions

### Helper auth

Use existing helper token contract.

### Operator auth

Reuse current admin auth/session semantics from `portal-http`.

### Command permissions

Start simple:

- admin can read all monitoring pages
- admin can execute commands

Later:

- separate read-only and operator roles

## Failure handling

### If helper unavailable

- node marked stale
- no commands dispatched
- operator sees `node_unreachable`

### If DB unavailable

- ingestor buffers minimally in memory
- API becomes degraded
- no fake healthy data served

### If dispatcher fails mid-command

- command remains `dispatched` with timeout window
- sweeper can reclassify to `timed_out`

## Recommended v1 language stance

Use Python for ingestor and API to stay aligned with existing control-plane.

That is not an ideological claim.
It is a coordination and operational simplicity claim.
