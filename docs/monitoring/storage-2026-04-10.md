# Monitoring Storage

Date: 2026-04-10

## Storage decision

Для v1 достаточно одного Postgres schema.

Не нужен отдельный TSDB на старте, потому что:

- объем пока умеренный
- нужны relational joins между `links`, `nodes`, `gateways`, `commands`, `incidents`
- existing control-plane already moves toward explicit Postgres-backed model

## Storage goals

- fast current status reads
- reliable event history
- durable command audit
- incident correlation
- cheap enough retention for operator debugging

## Current-state read path

UI should read current state from:

- `monitor_link_snapshots`
- joined with `monitor_links`
- joined with `monitor_nodes`

Не из event log.

## Event retention

### Recommended v1

- `monitor_link_snapshots`: current only
- `monitor_link_events`: 90 days hot retention
- `monitor_probes`: 30 days
- `monitor_commands`: 180 days
- `monitor_incidents`: indefinite

### Why

Operator incidents and commands are low-volume/high-value.
Raw link events are medium-value and can be pruned.

## Snapshot write model

Use upsert by `link_id`.

Rules:

- newer `observed_at` wins
- stale or duplicate older snapshots do not overwrite current row
- raw event still stored even if snapshot ignored

## Derived fields

Some fields should be computed centrally:

- `health_status`
- `stale_after`
- `incident candidate flags`

This avoids coupling UI semantics to raw helper payload quirks.

## Query patterns

### Fleet overview

Need:

- count by health
- count by node
- count by gateway
- active incidents

### Links table

Need filters:

- health
- state
- node
- gateway
- role
- stale
- flapping

### Link detail

Need:

- current snapshot
- recent events
- recent probes
- recent commands
- related incident state

## Why not store only raw JSON

Потому что тогда:

- фильтры станут дорогими
- UI будет завязан на payload shape
- alert rules будут сложными
- joins с edge/uplink model будут неудобными

JSON нужен как escape hatch, не как primary schema.
