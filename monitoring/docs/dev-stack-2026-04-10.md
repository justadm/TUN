## Services

### `postgres`

Stores:

- monitoring tables
- views
- future incidents and command audit

### `monitoring-api`

Current role:

- central API starter
- health endpoint
- SQL-backed `summary`
- SQL-backed `links`
- SQL-backed `nodes`
- SQL-backed `gateways`
- SQL-backed `link detail`
- SQL-backed `incidents`
- SQL-backed `commands`
- SQL-backed `commands audit`
- dev-mode app-facing `/v1/app/runtime/*`

### `monitoring-ingestor`

Current role:

- polling ingestion worker
- node source loading
- helper `links[]` polling
- `monitor_nodes` sync
- `monitor_links` sync
- `monitor_link_snapshots` sync
- normalized `monitor_link_events` writes on detected transitions
- passive `monitor_probes` writes from helper polling
- basic incident evaluator for `link_failed`, `link_stale`, `link_degraded`
- helper failure correlation into `node_unreachable`
- optional command dispatcher (`accepted -> dispatched -> succeeded/failed`)
- retry/backoff for failed command dispatches (env-configurable)
- status endpoint

## Local startup

1. copy `.env.example` to `.env`
2. copy `config/nodes.example.json` to `config/nodes.json`
3. run `make up` inside `monitoring/`

## Command e2e smoke

When helper action endpoints are available and dispatcher is enabled:

1. set `MONITORING_COMMAND_DISPATCH_ENABLED=true` in `.env`
2. restart stack (`make down && make up`)
3. run:
   - `make smoke-command LINK_ID=<link_id> COMMAND=reconnect`

## Current limitation

This stack is only a scaffold.

It does not yet:

- ingest helper SSE events
- write probes
- execute incident routing/escalation policy
- guarantee helper command endpoint availability on every node
- provide durable dispatch retries / backoff policy
- serve the full SQL-backed monitoring endpoint surface

It does give us:

- stable directory layout
- container boundaries
- environment contract
- reproducible local dev entrypoint
- initial SQL-backed `/v1/monitor/summary`
- initial SQL-backed `/v1/monitor/links`
- initial SQL-backed `/v1/monitor/nodes`
- initial SQL-backed `/v1/monitor/gateways`
- initial SQL-backed `/v1/monitor/links/{link_id}`
- initial SQL-backed `/v1/monitor/incidents`
- initial SQL-backed `/v1/monitor/incidents/{incident_id}`
- initial SQL-backed `/v1/monitor/commands`
- initial SQL-backed `/v1/monitor/commands/{command_id}`
- initial SQL-backed `/v1/monitor/commands/audit`
- initial SQL-backed `/v1/app/runtime/summary`
- initial SQL-backed `/v1/app/runtime/profiles/{connection_profile_id}`
- initial node inventory sync into `monitor_nodes`
- initial helper link polling into `monitor_links` and `monitor_link_snapshots`
- initial transition-derived writes into `monitor_link_events`
- initial passive probe writes into `monitor_probes`
- initial incident lifecycle for link and node reachability failures
- initial command queue and dispatch flow (env-gated)
- initial retry/backoff policy for command dispatch
