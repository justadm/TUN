# Monitoring DB Model

Date: 2026-04-10

## Goal

Эта схема нужна для центрального monitoring backend, который собирает telemetry от `runtime-helper` и отдает:

- operator/admin runtime views
- command audit
- incident views
- app-facing runtime summary through `JsTun`

## Design rules

### 1. Separate inventory, current state, and history

Нельзя смешивать:

- what exists
- what is true now
- what happened over time

Поэтому схема разделена на:

- inventory tables
- current snapshot tables
- append-only history tables

### 2. Keep runtime truth separate from product ownership

Monitoring domain знает про:

- nodes
- links
- sessions
- incidents
- commands

Product ownership comes from `JsTun` and should be attached explicitly, not inferred from random runtime fields.

### 3. Optimize for operator reads first

Main read patterns:

- fleet summary
- links table
- node detail
- gateway detail
- incident timeline

## Core tables

### `monitor_nodes`

Host inventory that exposes helper/runtime telemetry.

### `monitor_links`

Canonical managed link inventory.

### `monitor_link_subjects`

Binds runtime links to `JsTun` product entities:

- `account_id`
- `device_id`
- `connection_profile_id`

### `monitor_link_snapshots`

Current state only, one row per link.

### `monitor_link_events`

Append-only normalized link event log.

### `monitor_probes`

Synthetic and passive probe results.

### `monitor_commands`

Operator and automation command audit.

### `monitor_incidents`

Correlated active/resolved incidents.

### `monitor_incident_links`

Links affected by a given incident.

## Main query model

### Fleet overview

Read from:

- `monitor_link_snapshots`
- grouped by `health_status`, `node_id`, `gateway_id_selected`

### Link detail

Read from:

- `monitor_links`
- `monitor_link_snapshots`
- `monitor_link_events`
- `monitor_probes`
- `monitor_commands`
- `monitor_link_subjects`

### App runtime summary

Read from:

- `monitor_link_subjects`
- joined with `monitor_link_snapshots`

This is the reason product correlation needs to exist in the monitoring DB.

## Notes on future evolution

- If metric cardinality grows sharply, time-series data can later be moved into a dedicated metrics store.
- Current design intentionally keeps everything in Postgres for operational simplicity.
